10개월간 진행한 MSA 전환 프로젝트를 마무리하며, 프로젝트 리더(PL)로서 겪었던 기술적 결정과 조직 운영 경험을 정리합니다. 126개의 Jira 서브 티켓, 5명의 팀원, 그리고 수많은 배포 사이에서 배운 것들입니다.

## 전환을 결정한 배경

O2O 플랫폼 서비스는 하나의 Nest.js 모놀리식 서버(이하 앱 서버)에서 시작했습니다. 병원 접수, 예약, 검색, 알림, 결제 등 모든 도메인이 하나의 코드베이스에 공존했고, 서비스가 성장하면서 한계가 뚜렷해졌습니다.

- **배포 리스크 증가** — 접수 로직 한 줄 수정에도 전체 서비스 배포가 필요
- **코드 의존성 복잡화** — 도메인 간 경계가 모호해 side effect 빈번 발생
- **스케일링 비효율** — 특정 도메인의 트래픽 급증 시 전체 서버를 scale-out해야 함
- **Lambda 난립** — 단발성 기능들이 Lambda로 흩어져 관리 사각지대 발생

특히 앱 서버의 코드량이 늘어날수록 빌드 시간이 길어졌고, 신규 입사자의 온보딩 비용도 증가했습니다. "이 코드를 고치면 어디에 영향이 가는지" 아무도 자신있게 답하지 못하는 상황이 전환의 결정적 트리거였습니다.

## 프로젝트 설계 — Phase 1과 Phase 2

전환을 두 단계로 나누어 리스크를 관리했습니다.

```text
Phase 1: API 마이그레이션                    Phase 2: 데이터/인프라 분리
┌─────────────────────────┐                ┌─────────────────────────┐
│ - 앱 서버 API를 도메인    │                │ - DB 물리적 분리          │
│   서버로 이관             │                │ - 도메인 레벨 코드 이관    │
│ - 앱 서버 트래픽 차단     │                │ - Lambda → Consumer 전환  │
│ - 관리 서버 우선 컷오프   │                │ - AWS Batch → Argo Wflow  │
└─────────────────────────┘                └─────────────────────────┘
         ~5개월                                     ~5개월
```

Phase 1에서는 앱 서버의 API를 각 도메인 서버로 옮기고, 앱 서버로 가는 트래픽을 점진적으로 차단하는 것이 목표였습니다. Phase 2에서는 공유 데이터베이스를 물리적으로 분리하고, Lambda와 AWS Batch 같은 레거시 인프라를 정리하는 것이 목표였습니다.

### DDD 기반 바운디드 컨텍스트

팀 전체가 참여하는 Event Storming 워크숍을 통해 도메인을 식별했습니다.

```text
┌──────────┐    ┌──────────────┐    ┌──────────┐
│  사용자   │───→│  접수/예약    │───→│   병원   │
└──────────┘    └──────┬───────┘    └──────────┘
                       │
                ┌──────┴───────┐
                │     검색      │───→ 알림
                └──────┬───────┘
                       │
                ┌──────┴───────┐
                │     결제      │
                └──────────────┘
```

각 바운디드 컨텍스트에 **도메인 오너**를 지정하고, 컨텍스트 간 의존 방향을 명확히 정의했습니다. 접수 서비스가 가장 복잡한 도메인이었기에, Phase 1에서는 비교적 독립적인 관리 서버부터 컷오프하고 Phase 2에서 접수 서비스 리팩토링을 진행했습니다.

## 진행 과정 — API 전수조사부터 배포까지

### 모니터링 로그 기반 API 전수조사

전환의 첫 단계는 "지금 앱 서버에서 실제로 호출되는 API가 뭔지" 파악하는 것이었습니다. Datadog APM과 OpenSearch 로그를 기반으로 전체 API를 조사했습니다.

```text
[전수조사 프로세스]

Datadog APM 트레이스    OpenSearch 접근 로그
        │                      │
        └──────┬───────────────┘
               ▼
       API 엔드포인트 목록 추출
               │
               ▼
       호출 빈도/의존관계 분석
               │
               ▼
       도메인별 분류 + 우선순위 산정
               │
               ▼
       Jira 티켓 생성 (API별 1티켓)
```

이 조사를 통해 앱 서버에 존재하지만 더 이상 호출되지 않는 **죽은 API**도 상당수 발견했습니다. 이런 API는 마이그레이션 대상에서 제외하고 deprecated 처리했습니다.

### 티켓 관리와 업무 위임

API별로 Jira 티켓을 생성하고, 세부 작업은 서브 태스크로 분리했습니다. 총 **126개의 서브 티켓**(서브 태스크 제외)이 만들어졌습니다.

| 구분 | 설명 |
|------|------|
| 티켓 단위 | API 엔드포인트 1개 = 티켓 1개 |
| 서브 태스크 | 코드 이관, 테스트 작성, API 문서 갱신 등 |
| 담당 배정 | 각 스쿼드에 위임, 인력 부족 시 PL이 직접 처리 (단, PR 리뷰 필수) |
| 완료 기준 | 테스트 환경 검증 + 프로덕션 배포 + 모니터링 확인 |

PL로서 가장 신경 쓴 부분은 **티켓의 완료 기준을 명확히 하는 것**이었습니다. "코드 머지"가 아니라 "프로덕션 배포 후 모니터링까지 확인"을 완료 기준으로 잡았습니다.

### 배포 프로세스

빠른 전환을 위해 경량화된 배포 프로세스를 도입했습니다.

```text
dev 브랜치 → 로컬 테스트 → test 환경 배포 → QA 검증 → production 배포
                                                          │
                                                    모니터링 확인
                                                    (Datadog APM)
```

마이그레이션 특성상 기능 변경이 아닌 코드 이동이 대부분이었기에, 로컬 테스트와 테스트 환경 검증에 집중하고 배포 사이클을 짧게 가져갔습니다.

## 기술적 결정들

### 서비스 간 통신: REST + Kafka

동기 통신은 REST, 비동기 통신은 Kafka로 분리했습니다. 도메인 이벤트가 발생하면 Kafka로 발행하고, 관심 있는 서비스가 구독하는 구조입니다.

```typescript
// 이벤트 발행 예시 — 예약 생성 시 알림 서비스에 이벤트 전달
@Injectable()
export class ReservationService {
  constructor(
    private readonly kafkaProducer: KafkaProducerService,
    private readonly reservationRepo: ReservationRepository,
  ) {}

  async createReservation(dto: CreateReservationDto) {
    const reservation = await this.reservationRepo.create(dto);

    // 이벤트 발행 — 알림 서비스, 통계 서비스 등이 구독
    await this.kafkaProducer.emit('reservation.created', {
      reservationId: reservation.id,
      userId: dto.userId,
      hospitalId: dto.hospitalId,
      timestamp: new Date().toISOString(),
    });

    return reservation;
  }
}
```

```typescript
// 컨슈머 측 멱등성 보장
@Injectable()
export class NotificationConsumer {
  constructor(
    private readonly processedEventRepo: ProcessedEventRepository,
    private readonly notificationService: NotificationService,
  ) {}

  @KafkaListener('reservation.created')
  async handleReservationCreated(message: ReservationCreatedEvent) {
    // 중복 메시지 방어 — eventId 기반 멱등성 체크
    const alreadyProcessed = await this.processedEventRepo.exists(
      message.eventId,
    );
    if (alreadyProcessed) return;

    await this.notificationService.sendReservationConfirm(message);
    await this.processedEventRepo.save(message.eventId);
  }
}
```

이벤트 기반 통신에서 가장 주의한 점은 **멱등성 보장**이었습니다. 네트워크 이슈로 메시지가 중복 전달될 수 있기에, 컨슈머 측에서 `eventId` 기반으로 중복을 걸러내도록 설계했습니다.

### 데이터 분리 전략

Phase 2의 핵심인 데이터베이스 물리적 분리는 가장 고민이 많았던 부분입니다.

```text
[Before]                           [After]
┌─────────────────┐               ┌──────────┐  ┌──────────┐
│   Shared MongoDB │               │ 접수 DB   │  │ 예약 DB   │
│                  │       →       └──────────┘  └──────────┘
│ 접수, 예약, 병원, │               ┌──────────┐  ┌──────────┐
│ 사용자, 결제 ...  │               │ 병원 DB   │  │ 사용자 DB │
└─────────────────┘               └──────────┘  └──────────┘
```

서비스별 독립 DB를 원칙으로 하되, 데이터 정합성이 중요한 경우에는 **Saga 패턴**을 적용했습니다. 예를 들어, 접수 취소 시 결제 환불이 함께 이루어져야 하는 경우입니다.

```typescript
// Choreography Saga — 이벤트 체인으로 트랜잭션 관리
// 1. 접수 서비스: 접수 취소 → 'reception.cancelled' 이벤트 발행
// 2. 결제 서비스: 이벤트 수신 → 환불 처리 → 'payment.refunded' 발행
// 3. 알림 서비스: 이벤트 수신 → 사용자에게 취소 알림 발송
```

### 인프라 전환: Lambda 제거와 AWS Batch 교체

모놀리식 시절 단발성 기능들이 AWS Lambda로 구현되어 있었습니다. 이것들을 Kafka Consumer로 통합 마이그레이션했습니다.

| 항목 | Before (Lambda) | After (Kafka Consumer) |
|------|-----------------|----------------------|
| 트리거 | CloudWatch Event, API Gateway | Kafka 토픽 구독 |
| 모니터링 | CloudWatch Logs (분산) | 중앙 집중 로깅 (Loki) |
| 배포 | Serverless Framework | Kubernetes Deployment |
| 디버깅 | 로컬 재현 어려움 | 로컬 Kafka로 재현 가능 |
| 콜드 스타트 | 있음 (VPC 내 최대 10초) | 없음 (상시 구동) |

AWS Batch로 실행하던 배치 작업은 **Argo Workflow**로 전환했습니다.

```yaml
# Argo Workflow 예시 — 일일 통계 집계
apiVersion: argoproj.io/v1alpha1
kind: CronWorkflow
metadata:
  name: daily-stats-aggregation
spec:
  schedule: "0 2 * * *"
  timezone: Asia/Seoul
  workflowSpec:
    entrypoint: aggregate
    templates:
      - name: aggregate
        container:
          image: stats-worker:latest
          command: ["node", "dist/batch/daily-stats.js"]
          resources:
            requests:
              memory: "512Mi"
              cpu: "250m"
```

Argo Workflow로 전환한 가장 큰 이유는 **Kubernetes 네이티브**라는 점이었습니다. 기존 인프라와 동일한 모니터링/로깅 체계를 그대로 사용할 수 있었고, DAG 기반 워크플로우 정의로 복잡한 배치 의존성도 깔끔하게 관리할 수 있었습니다.

## PL로서 배운 것

### 일정 관리 — 126개 티켓, 10개월

프로젝트 일정을 100% 달성할 수 있었던 핵심 요인은 세 가지입니다.

1. **버퍼를 포함한 현실적 일정 수립** — 각 태스크에 20% 버퍼를 적용했습니다. 마이그레이션은 "단순 코드 이동"처럼 보이지만, 실제로는 레거시 코드 이해, 테스트 작성, 엣지 케이스 처리에 예상보다 많은 시간이 걸립니다.

2. **주간 단위 마일스톤** — 큰 Phase를 주간 단위로 쪼개 진행률을 추적했습니다. 2주 이상 지연되는 티켓은 즉시 리스크로 에스컬레이션했습니다.

3. **블로커 조기 식별** — 매일 스탠드업에서 블로커를 공유하고, PL이 직접 해결하거나 다른 팀에 에스컬레이션했습니다.

### 팀 협업과 기술적 합의

기술적 결정만큼 중요한 것이 팀의 합의였습니다. 주요 아키텍처 결정은 **ADR(Architecture Decision Record)**로 문서화하여, 결정의 맥락과 트레이드오프를 기록했습니다.

스쿼드에 업무를 위임할 때는 "왜 이렇게 분리하는지"를 설명하는 데 시간을 투자했습니다. 단순히 티켓을 할당하는 것과 맥락을 공유하고 할당하는 것은 결과물의 품질이 확연히 달랐습니다.

### 모니터링의 중요성

프로젝트를 통해 가장 뼈저리게 느낀 것은 **배포 후 모니터링**의 중요성이었습니다. 초반에는 "에러 로그가 없으면 성공"이라고 판단했는데, 실제로는 에러 없이도 문제가 발생하는 경우가 많았습니다.

```text
[모니터링 체크리스트 — 마이그레이션 배포 후]

□ Datadog APM에서 신규 엔드포인트 응답시간 확인
□ 기존 앱 서버 엔드포인트 트래픽 감소 확인
□ Kafka Consumer lag 정상 범위 확인
□ 외부 연동 API 응답 상태 확인   ← 놓치기 쉬운 포인트
□ 에러율뿐 아니라 비즈니스 메트릭(접수 건수 등) 비교
```

특히 **외부 API 모니터링**은 초반에 간과했던 부분입니다. 마이그레이션 자체는 성공했지만, 외부 연동 부분에서 타임아웃 설정이 달라 간헐적 실패가 발생한 적이 있었습니다.

## 결과 — 수치 기반 성과

| 지표 | Before (모놀리식) | After (MSA) |
|------|-------------------|-------------|
| 배포 단위 | 전체 서비스 일괄 | 도메인별 독립 배포 |
| 배포 빈도 | 주 2-3회 | 주 8-10회 (약 3배) |
| 평균 배포 시간 | 15분 | 5분 |
| 장애 영향 범위 | 전체 서비스 | 해당 도메인만 격리 |
| 프로젝트 규모 | - | 126개 서브 티켓, 10개월 |
| 앱 서버 트래픽 | 100% | 점진적 감소 → 최종 컷오프 |

앱 서버의 트래픽은 API 이관이 진행될수록 점진적으로 감소했고, Phase 1 완료 시점에 관리 서버 컷오프를 달성했습니다.

## 팀 회고 — 5명이 돌아본 10개월

프로젝트 종료 후 팀원 5명이 함께 회고를 진행했습니다.

### 잘된 점

- **레거시 코드 개선 기회** — 마이그레이션 과정에서 기존의 문제 있는 코드를 자연스럽게 리팩토링할 수 있었습니다. 한 시니어 개발자는 "단순 이관이 아니라 코드 품질을 높이는 기회로 활용한 것이 좋았다"고 평가했습니다.
- **타 도메인 학습** — 자신의 담당이 아닌 도메인 코드를 이관하면서, 서비스 전체에 대한 이해도가 높아졌습니다.
- **체계적인 티켓 관리** — API별로 티켓을 잘게 쪼갠 덕분에 진행 상황이 투명하게 공유되었고, 병렬 작업이 가능했습니다.

### 아쉬운 점

- **QA 단계에서 발견된 버그** — 이미지 관련 처리 등 일부 엣지 케이스가 QA 팀에서 발견되었습니다. 마이그레이션 코드에 대한 자동화 테스트가 부족했던 결과였습니다.
- **이관된 코드 이해도** — 다른 팀원이 작성한 마이그레이션 코드를 이후에 유지보수할 때, 원본 코드의 맥락을 모르면 이해가 어려운 경우가 있었습니다.

### 배운 점

- **배포 후 모니터링은 "에러 없음" 이상이어야 한다** — 에러 로그가 없어도 비즈니스 메트릭(접수 건수, 예약 성공률)을 함께 확인해야 진짜 정상인지 판단할 수 있습니다.
- **외부 API 모니터링은 별도로 필요하다** — 내부 서비스 간 통신만 모니터링하면 외부 연동 실패를 놓칩니다.
- **마이그레이션 작업에는 일반 개발보다 더 꼼꼼한 테스트가 필요하다** — "동일한 동작을 보장"한다는 것 자체가 높은 수준의 테스트 커버리지를 요구합니다.

## 회고 및 향후 계획

### 돌아보며

10개월간의 프로젝트를 한마디로 정리하면, **"기술적 결정보다 조직적 결정이 더 어려웠다"**입니다. 어떤 패턴을 쓸지, 어떤 도구를 도입할지는 기술적 판단으로 결정할 수 있지만, 누가 어떤 도메인을 맡을지, 스쿼드 인력이 부족할 때 어떻게 조율할지는 사람의 문제입니다.

콘웨이 법칙이 말하듯, 아키텍처는 결국 조직 구조를 반영합니다. MSA 전환은 코드를 나누는 것이 아니라 **팀의 일하는 방식을 바꾸는 것**이었습니다.

### 향후 계획

- **MongoDB/Redis 클러스터 분리** — 현재 공유 클러스터를 도메인별로 물리적 분리 예정
- **서비스 메시 도입 검토** — 서비스 간 통신의 관측성과 보안을 강화하기 위해 Istio 도입 검토 중
- **성능 최적화** — 마이그레이션 완료 후 각 도메인 서비스의 쿼리 최적화 및 캐시 전략 고도화
